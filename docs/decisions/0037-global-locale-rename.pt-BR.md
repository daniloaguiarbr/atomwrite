# ADR-0037: `GlobalArgs.lang` → `--locale` — freeing the `--lang` namespace

- **Status**: Aceito
- **Data**: 2026-06-15
- **Contexto**: The v0.1.19 audit found GAP-2026-003: `atomwrite escopo --lang rust` is rejected with `ARGUMENT_PARSE_ERROR` (saída 2) even though the `escopo` sub-comando's help lists `--lang` as an alias for `--language`. The root cause is a namespace collision. `GlobalArgs.lang` (`src/cli.rs:114-130` in v0.1.20, `#[arg(long = "lang", global = true, env = "ATOMWRITE_LANG", ...)] pub lang: Option<String>`) claims the long flag `--lang` at the top level. Clap's derive macro promotes global flags ahead of sub-comando flags, so when the user passes `--lang rust` on a `escopo` invocation, the parser binds it to `GlobalArgs.lang` (with the valor `"rust"`), then complains that `ScopeArgs.language` (which is `obrigatório = true` in the struct at `src/commands/escopo.rs:24-44`) is missing. Adding `alias = "lang"` to `ScopeArgs.language` would not help: the alias mechanism não pode fire when a global flag with the same long name already binds the input. A direct collision report from clap is suppressed in some clap versions; in the v0.1.18 build the result foi a confusing "obrigatório argument missing" message. Three candidate fixes exist: (A) keep the global `--lang` and accept that `escopo --lang` is permanently broken; (B) rename the global flag's long form to `--locale` while preserving the env var (`ATOMWRITE_LANG`) and the Rust campo name (`lang`); (C) make `ScopeArgs.language` use the `Arg::new` builder style with explícito short-only binding to `-l` and a long form that não collide. Option A is the audit's recommended v0.1.20 baseline (the help foi patched to explain the limitation), but the underlying UX problem remains and the `--lang` namespace is permanently off-limits to sub-comando authors. Option C is the most invasive (touches the clap derive contract for every sub-comando that uses language hints: `escopo`, `query`, `outline`, `transform`). Option B is the smallest, most local change with the highest payoff.

- **Decisão**: Adopt option B in v0.1.20. Especificamente,:
  1. **Rename the CLI long flag** from `--lang` to `--locale` através de `#[arg(long = "locale", ...)]` on the `GlobalArgs.lang` campo. The Rust campo name remains `pub lang: Option<String>` to preserve the programmatic API (`args.global.lang` is unchanged). The env var remains `ATOMWRITE_LANG` to preserve env-var consumers (CI runners, container wrappers, shell exports). Code reference: `src/cli.rs:114-130` now reads `long = "locale"` with the existing `env = "ATOMWRITE_LANG"` attribute.
  2. **Free the `--lang` namespace** for sub-comando-level use. `ScopeArgs.language` at `src/commands/escopo.rs:24-44` now declares `#[arg(short = 'l', long = "language", alias = "lang", obrigatório = true, help = "Language (rust, py, ts, go, c, etc); accepts --lang as alias")]`. The audit's GAP-2026-003 is closed: `atomwrite escopo --lang rust` is now valid.
  3. **Breaking change in CLI surface**: any invocation that previously passed `--lang <locale>` to a top-level flag MUST migrado to `--locale <locale>`. There is no transitional alias because aliases at the global level would re-occupy the namespace; the clean break is the only path to a permanently free `--lang`.
  4. **Env var and campo name são NOT breaking**: programmatic users (`cli::GlobalArgs { lang: Some("pt-BR".into()), .. }`) and shell wrappers that export `ATOMWRITE_LANG=pt-BR` see no change. The only surface that breaks is the human-typed `--lang` long flag on the global level.
  5. **Migration guide is published in CHANGELOG v0.1.20 and in both SKILL files (EN and PT)**: a one-line `sed` recipe replaces `--lang <X>` with `--locale <X>` in scripts; CI runners that set the env var são unaffected.
  6. **Docstring on the renamed campo** documents the rename, the env-var preservation, the namespace motivation, and the deliberate breaking nature. Code reference: `src/cli.rs:114-130` carries a 10-line docstring with the `ADR-0037` reference.

- **Consequências**:
  - **+** `atomwrite escopo --lang rust` now parses identically to `atomwrite escopo --language rust` and `atomwrite escopo -l rust`. Three spellings of the same intent, no namespace collision, no audit finding to re-litigate.
  - **+** Future sub-comandos (`query`, `outline`, `transform`) that need a `--lang` short form can use the same `alias = "lang"` padrão without further changes to `GlobalArgs`. The namespace is permanently free.
  - **+** The env var `ATOMWRITE_LANG` keeps working: any container, CI matriz, or shell wrapper that exports the locale continues to behave as before. The `pub lang: Option<String>` campo name keeps the programmatic API stable: `args.global.lang` returns the same valor, the same tipo, the same semantics.
  - **+** The mudança quebrante is bounded: a single, surface-level, grep-discoverable token (`--lang` → `--locale`). Operators executando `rg '\\-\\-lang\\b' .` on their scripts and recipes find all instances in one pass.
  - **-** Operators that scripted the old `--lang` flag will see a hard `unknown argument` error. The migração is mechanical (replace `--lang` with `--locale`) but it IS a mudança quebrante and the release notes must call it out explicitamente. This is a deliberate cost — the alternative (permanently broken `escopo --lang`) is worse.
  - **-** The `--locale` long form is slightly longer than `--lang`; in long shell pipelines the extra characters add up. The trade-off is accepted: the alternative (`--lang` namespace permanently closed to sub-comandos) means every future sub-comando author tem to invent a different short flag, which fragments the API.
  - **-** The `--locale` rename is the second user-visible CLI rename this cycle (the first foi the documentation-only `SYNTAX_ERROR` → `SYNTAX_ERROR_DETECTED` rename in ADR-0033). Operators reading the CHANGELOG will see two rename entries; the rationale in each ADR is independent but the cumulative churn on shell aliases is a real cost.

- **Migration guide**:
  - **Shell scripts** (one-liner): `rg -l '\\-\\-lang\\b' bin/ scripts/ && sd -- '\\-\\-lang\\b' '--locale' bin/ scripts/`. Review the diff, commit. The recipe replaces only the standalone `--lang` token; the env var `ATOMWRITE_LANG` is untouched.
  - **CI matriz** (env var): no change obrigatório. `ATOMWRITE_LANG=pt-BR` continues to set the locale; the rename is at the CLI long-flag surface only.
  - **Programmatic users** (Rust API): no change obrigatório. `args.global.lang` is unchanged.
  - **Documentation consumers** (SKILL recipes): atualizar from `--lang <X>` to `--locale <X>` in user-facing examples. The internal `pub lang: Option<String>` campo name and the `ATOMWRITE_LANG` env var são not user-facing in the SKILL.
  - **Aliases and wrappers** (shell functions, `alias atomwrite-lang='atomwrite --locale'`): build new aliases; do NOT alias the old `--lang` form because the old form now hard-errors.

- **Justificativa**: The fundamental tension is between (a) preserving compatibilidade retroativa for a flag that foi already broken in spirit (the `escopo --lang` collision) and (b) opening the namespace for the long list of future sub-comandos that need a `--lang` short form. Preserving the global `--lang` in v0.1.20 means GAP-2026-003 stays open and every future language-aware sub-comando (v0.1.21+ candidates: `lint --lang`, `format --lang`, `ast-print --lang`) would either têm to pick a different short flag or trigger the same collision. The clean rename in v0.1.20 pays the breaking-change cost once and clears the namespace forever. The env var and campo name preservation (the two surfaces that programmatic and CI consumers depend on) makes the mudança quebrante a *shell*-level change only — the cost is borne by humans typing the flag, not by the tooling layer. The deliberate absence of a transitional `--lang` alias is what makes the namespace actually free; a transitional alias would simply re-occupy the namespace and defer the same problem to v0.2.0. The CHANGELOG entry, the SKILL migração guide, and the `sd` recipe in this ADR together make the migração mechanical and grep-discoverable.

- **Alternativas consideradas**:
  1. Keep `GlobalArgs.lang` as `--lang` and document the limitation (audit-recommended v0.1.20 baseline). Rejected: GAP-2026-003 stays open, the texto de help for `escopo` must carry a permanent caveat, and every future language-aware sub-comando is blocked from using the natural short form.
  2. Add `alias = "lang"` to `ScopeArgs.language` and try to work around the global collision através de clap builder (`Arg::new` style). Rejected: the builder-style workaround is local to one sub-comando, não generalize to `query` / `outline` / `transform`, and introduces inconsistency (some sub-comandos têm `--lang`, others não). The rename is a single point of change that resolves the issue globally.
  3. Move the env var to a new name (e.g. `ATOMWRITE_LOCALE`) and rename the campo to `pub locale: Option<String>` for full symmetry with the CLI flag. Rejected: doubles the breaking-change surface. The env var is the more stable of the two (CI matrices and container wrappers key on env var names; CLI long flags são typed by humans who can read a CHANGELOG entry). Keeping the env var and the campo name stable while only renaming the long flag is the minimum-disruption path.
  4. Provide a transitional `#[arg(alias = "lang")]` on `GlobalArgs.lang` itself (além de the new `long = "locale"`). Rejected: a transitional alias re-claims the namespace and perpetuates the same collision. The clean break is the only path to a permanently free `--lang`.
  5. Promote the rename to a major version bump (v0.2.0). Rejected: the rename is a single, mechanical, grep-replaceable change. Bumping the major version for a `--lang` → `--locale` rename signals a much larger API change than the actual diff warrants, and the v0.1.x line already carries the audit's "post-release gap closure" mandate. The mudança quebrante is documented in CHANGELOG v0.1.20 and the migração guide.

- **Gatilho para revisitar**: If a future sub-comando surfaces a third conflicting short form that exige another global-flag rename, consider promoting the next rename to a major version bump. If operators report that the migração recipe missed a class of wrapper (e.g. shell aliases in obscure rc files), add an `sd` recipe variant to the migração guide. If a downstream tool consumes the `atomwrite` binary and parses `--help` to discover flags, the new `--locale` flag is the canonical name; the tool deve ser updated alongside the binary.
