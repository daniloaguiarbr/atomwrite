# ADR-0038: Backup cumprido deleta — delete after success is the new default

- **Status**: Accepted
- **Date**: 2026-06-17
- **Context**: The v0.1.20 audit surfaced 11 GAP-2026 closures and 4 ADRs. A post-release sweep of consumer feedback identified 3 follow-on gaps that compose release v0.1.21. GAP-2026-014 v2 is the most consequential: `cleanup_old_backups_in` (atomic.rs:833-858) prunes by count, which means backups live indefinitely until 5 newer ones push them out. Every successful `--backup` operation leaves persistent trash on disk; CI scripts that run `fd '*.bak.*' . | wc -l` see growing counts proportional to write volume. The pre-v0.1.21 behavior inherited a VCS-era retention model where the backup ring is the artifact of record. For an editor atómico that is a mis-fit: the editor is not the VCS, the backup is a transitory safety net for one operation, not an audit log. The 2026-06-15 c24-framework34.html incident (ADR-0035) showed the value of explicit pre-write backup, but the same incident also showed the cost of the implicit post-write audit log: no operator inspects `.bak.*` siblings after success, the files accumulate, and the only consumer is the `fd` walker that surfaces them in every `eza --git` listing. The audit also found that `edit` and `rollback` hardcode `backup: false` in `AtomicWriteOptions` (edit.rs:139, edit.rs:393, rollback.rs:108), violating the API parity principle established by `write` and `replace`. The single decision that addresses both findings is: backups are deleted after success by default, and the parity hole is closed by exposing the same `--backup` flag to `edit` and `rollback` with the same post-success deletion semantics.

- **Decision**: Adopt a two-part rule for v0.1.21.
  1. **`keep_backup: bool` on `AtomicWriteOptions`, default `false`.** Every site that constructs an `AtomicWriteOptions` with `backup: Some(path)` gets a paired `keep_backup: opts.keep_backup || <user-opt-in>`. After `atomic_write_inner` returns success, if `keep_backup == false`, a new helper `delete_backup_quietly(&bp)` removes the backup file. On error, the backup is preserved as before. The helper is idempotent: `ErrorKind::NotFound` is mapped to `Ok(())` and emits `tracing::debug!`; non-NotFound errors emit `tracing::warn!` and the caller proceeds (cleanup is logged, not propagated). The user-facing opt-in is the new flag `--keep-backup`, accepted on 6 subcommands: `write`, `edit`, `replace`, `rollback`, `apply`, `batch`.
  2. **Close the `--backup` API parity hole.** `EditArgs` gains `backup: bool`, `retention: u8`, and `keep_backup: bool`. `RollbackArgs` gains `backup: bool` and `keep_backup: bool`. The 3 hardcoded `backup: false` sites are replaced with `args.backup`. `ReplaceArgs`, `ApplyArgs`, and `BatchArgs` gain `keep_backup: bool`. Subcommand parity for `--backup` is now 4/4 (write, edit, replace, rollback). `apply` propagates `args.keep_backup` to the inner `atomic_write`; `batch` propagates `--keep-backup` to every `write`/`edit`/`replace` op in the NDJSON manifest.

- **Consequences**:
  - **+** Post-success `fd '*.bak.*' . | wc -l` reports 0 on a clean run. The audit metric for GAP-2026-014 v2 (`Backups órfãos após sucesso: 0`) is now structurally enforced by the atomic layer, not by operator vigilance.
  - **+** `edit` and `rollback` now expose `--backup` to match `write` and `replace`. Users who scripted `--backup` on `edit` and were silently ignored now get a real call. The 3 hardcoded `backup: false` sites are gone; the `rules-rust-cli-com-clap-armadilhas-validacao` invariant (no derive struct field without consumption) is restored.
  - **+** `keep_backup: true` is an explicit operator decision. CI scripts that depend on the backup as forensic evidence pass `--keep-backup` and the behavior is mechanical, not depending on which subcommand or which flag combination was used.
  - **+** Failure path is unchanged: backups survive on any non-Ok return from `atomic_write_inner`. The new helper only runs on the success path, so the c24-framework34.html class of incident is unaffected.
  - **-** Breaking change for any operator that relied on `fd '*.bak.*' .` finding the post-success backup. The migration is `add --keep-backup` to the script. Documented in `docs/MIGRATION-v0.1.20-to-v0.1.21.md`.
  - **-** The `delete_backup_quietly` helper is exposed to a small set of failure modes (EBUSY, EACCES, EROFS) that map to `tracing::warn!` and silent success. An operator who wants a hard failure on cleanup can wrap the atomic call in a script-level check. The alternative — propagating the warning as an error — would create a worse UX (the write succeeded, the operator now has to handle a non-zero exit that says "could not delete the file we just told you to delete").
  - **-** 6 subcommands gain the `--keep-backup` flag, expanding the help surface by 6 lines. The flag is opt-in (default off) so power users who want the old behavior add the flag once and forget; new users see the simpler default.

- **Rationale**: The 2026-06-15 audit (ADR-0035) established that intention guards are layered, not binary. The same logic applies to backup retention: the operator who wants a forensic audit trail is not the operator who runs 1000 `--backup` writes per day. The default should match the common case (transient safety net, no audit) and the opt-in should match the rare case (explicit forensic). Deletion-after-success is the cheaper invariant; the old `cleanup_old_backups_in` count-based prune was a holdover from a project context where the backup WAS the audit log (no VCS). With the VCS in place, the backup is redundant by the time the next write happens, and the cost (one extra file per write, scanned on every `eza --git` and every `fd` walk) outweighs the benefit. The 6 subcommand opt-in is the smallest change that closes the API parity hole (GAP-2026-013 Problem C) without making any subcommand a regression for users who don't need backups. The helper idempotence and the `tracing::warn!` on non-NotFound errors preserve the L4 "the operation completed, the cleanup is advisory" principle that ADR-0028 established for the WAL sidecar.

- **Alternatives considered**:
  1. **Keep `cleanup_old_backups_in` count-based prune, add `--keep-backup` only.** Rejected: the count-based prune still leaves the first 5 backups forever and the 6th-and-after only on a 5-more-writes trigger. A user who writes 4 times never sees any cleanup. The metric (`Backups órfãos após sucesso: 0`) requires deletion-on-success, not deletion-on-overflow.
  2. **Add a separate `prune-backups` subcommand and let the operator run it on a schedule.** Rejected: introduces a daemon-adjacent operational burden (schedule, monitor, alarm) for a problem that can be solved inline. The `rules-rust-cli-one-shot §Daemon Proibido` invariant forbids anything that would require a long-running process for cleanup.
  3. **Age-based prune: `--max-backup-age <SECONDS>`.** Rejected: a 0-second age is equivalent to "delete immediately" (this ADR); a non-zero age reintroduces the "when do they go away" question that this ADR resolves by making the answer "after the write that created them succeeds". Subsumed by deletion-on-success.
  4. **Force `--backup` to imply `--keep-backup` (no opt-out).** Rejected: violates the project's "every flag is opt-in" convention. The 2026-06-15 incident is the precedent for adding safety flags, not the precedent for removing operator choice.

- **Trigger to revisit**: If a user reports that a backup was needed after success and they did not pass `--keep-backup` (e.g. a forensic timeline analysis that requires the post-success state to be reversible), revisit the default. If the `delete_backup_quietly` failure rate on real workloads exceeds 0.1% (measured via the new `tracing::warn!` on non-NotFound), add a `--strict-cleanup` flag that propagates cleanup errors as non-zero exit. If a future subcommand (e.g. `safe-write`, `move`) needs the same behavior, extend the flag list rather than introducing a subcommand-specific opt-in.

## Addendum (v0.1.22 — revisit of Alternative 2)

The v0.1.22 audit identified a context this ADR did not anticipate: **legacy `.bak.YYYYMMDD_HHMMSS` siblings created under v0.1.20 retention semantics**. Under v0.1.20, `cleanup_old_backups_in` retained the 5 most recent backups per file; an operator who upgraded from v0.1.20 to v0.1.21 inherits one `.bak.<timestamp>` sibling per `--backup` write they ever issued under the old retention semantics. The deletion-after-success invariant in the body of this ADR addresses the **forward-facing** problem (new writes do not leave persistent trash) but leaves the **backward-facing** problem (legacy v0.1.20 backups persist) without a removal path.

Alternative 2 of this ADR ("Add a separate `prune-backups` subcommand and let the operator run it on a schedule") was rejected at the time on the grounds that it introduced a daemon-adjacent operational burden. v0.1.22 revisits that decision with a narrower scope: the `prune-backups` subcommand is **explicit** (the operator runs it on purpose, not as a side effect of another subcommand), **safe** (defaults to `--dry-run true` to prevent accidental data loss), and **composable** (uses the same `--workspace` jail and the same NDJSON envelope as the rest of the CLI). The operator-driven one-shot invocation satisfies the `rules-rust-cli-one-shot §Daemon Proibido` invariant that was the original reason for rejection — there is no schedule, no background process, no implicit cleanup.

The full rationale for the v0.1.22 subcommand is documented in ADR-0040 (`docs/decisions/0040-prune-backups-subcommand.md`). The two ADRs are complementary: ADR-0038 governs the lifecycle of backups created by new writes (deleted after success by default); ADR-0040 governs the explicit cleanup of pre-v0.1.21 backup siblings.

No retroactive automatic cleanup is introduced. An operator who upgrades from v0.1.20 to v0.1.22 must run `prune-backups --max-age 0 --dry-run false` explicitly to clean up legacy backups. This is intentional: the v0.1.20 backups were created under different semantics (implicit count-based retention), and silently deleting them on `init` or on first write of the new version would be a data-loss bug, not a cleanup.
