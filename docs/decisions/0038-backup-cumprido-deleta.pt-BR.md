# ADR-0038: Backup cumprido deleta — delete after success is the new padrão

- **Status**: Aceito
- **Data**: 2026-06-17
- **Contexto**: The v0.1.20 audit surfaced 11 GAP-2026 closures and 4 ADRs. A post-release sweep of consumer feedback identified 3 follow-on gaps that compose release v0.1.21. GAP-2026-014 v2 is the most consequential: `cleanup_old_backups_in` (atomic.rs:833-858) prunes by count, which means backups live indefinitely until 5 newer ones push them out. Every successful `--backup` operation leaves persistent trash on disk; CI scripts that executar `fd '*.bak.*' . | wc -l` see growing counts proportional to write volume. The pre-v0.1.21 behavior inherited a VCS-era retention model where the backup ring is the artifact of record. For an editor atómico that is a mis-fit: the editor is not the VCS, the backup is a transitory safety net for one operation, not an audit log. The 2026-06-15 c24-framework34.html incident (ADR-0035) showed the valor of explícito pre-write backup, but the same incident also showed the cost of the implícito post-write audit log: no operator inspects `.bak.*` siblings after success, the files accumulate, and the only consumer is the `fd` walker that surfaces them in every `eza --git` listing. The audit also found that `edit` and `rollback` hardcode `backup: false` in `AtomicWriteOptions` (edit.rs:139, edit.rs:393, rollback.rs:108), violating the API parity principle established by `write` and `replace`. The single decision that endereça both findings is: backups são deleted after success by padrão, and the parity hole is closed by exposing the same `--backup` flag to `edit` and `rollback` with the same post-success deletion semantics.

- **Decisão**: Adopt a two-part regra for v0.1.21.
  1. **`keep_backup: bool` on `AtomicWriteOptions`, padrão `false`.** Every site that constructs an `AtomicWriteOptions` with `backup: Some(path)` gets a paired `keep_backup: opts.keep_backup || <user-opt-in>`. After `atomic_write_inner` returns success, if `keep_backup == false`, a new helper `delete_backup_quietly(&bp)` removes the backup file. On error, the backup is preserved as before. The helper is idempotent: `ErrorKind::NotFound` is mapped to `Ok(())` and emits `tracing::debug!`; non-NotFound errors emit `tracing::warn!` and the caller proceeds (cleanup is logged, not propagated). The user-facing opt-in is the new flag `--keep-backup`, accepted on 6 sub-comandos: `write`, `edit`, `replace`, `rollback`, `apply`, `batch`.
  2. **Close the `--backup` API parity hole.** `EditArgs` gains `backup: bool`, `retention: u8`, and `keep_backup: bool`. `RollbackArgs` gains `backup: bool` and `keep_backup: bool`. The 3 hardcoded `backup: false` sites são replaced with `args.backup`. `ReplaceArgs`, `ApplyArgs`, and `BatchArgs` gain `keep_backup: bool`. Subcommand parity for `--backup` is now 4/4 (write, edit, replace, rollback). `apply` propagates `args.keep_backup` to the inner `atomic_write`; `batch` propagates `--keep-backup` to every `write`/`edit`/`replace` op in the NDJSON manifest.

- **Consequências**:
  - **+** Post-success `fd '*.bak.*' . | wc -l` reports 0 on a clean executar. The audit metric for GAP-2026-014 v2 (`Backups órfãos após sucesso: 0`) is now structurally enforced by the atomic layer, not by operator vigilance.
  - **+** `edit` and `rollback` now expose `--backup` to match `write` and `replace`. Users who scripted `--backup` on `edit` and foram silently ignored now get a real call. The 3 hardcoded `backup: false` sites são gone; the `regras-rust-cli-com-clap-armadilhas-validacao` invariante (no derive struct campo without consumption) is restored.
  - **+** `keep_backup: true` is an explícito operator decision. CI scripts that depend on the backup as forensic evidence pass `--keep-backup` and the behavior is mechanical, not depending on which sub-comando or which flag combination foi used.
  - **+** Failure path is unchanged: backups survive on any non-Ok return from `atomic_write_inner`. The new helper only runs on the success path, so the c24-framework34.html class of incident is unaffected.
  - **-** Breaking change for any operator that relied on `fd '*.bak.*' .` finding the post-success backup. The migração is `add --keep-backup` to the script. Documented in `docs/MIGRATION-v0.1.20-to-v0.1.21.md`.
  - **-** The `delete_backup_quietly` helper is exposed to a small set of failure modes (EBUSY, EACCES, EROFS) that map to `tracing::warn!` and silent success. An operator who wants a hard failure on cleanup can wrap the atomic call in a script-level check. The alternative — propagating the warning as an error — would create a worse UX (the write succeeded, the operator now tem to handle a non-zero saída that says "could not delete the file we just told you to delete").
  - **-** 6 sub-comandos gain the `--keep-backup` flag, expanding the help surface by 6 lines. The flag is opt-in (padrão off) so power users who want the old behavior add the flag once and forget; new users see the simpler padrão.

- **Justificativa**: The 2026-06-15 audit (ADR-0035) established that intention guards são layered, not binary. The same logic aplica to backup retention: the operator who wants a forensic audit trail is not the operator who runs 1000 `--backup` writes por day. The padrão should match the common case (transitório safety net, no audit) and the opt-in should match the rare case (explícito forensic). Deletion-after-success is the cheaper invariante; the old `cleanup_old_backups_in` count-based prune foi a holdover from a project context where the backup WAS the audit log (no VCS). With the VCS in place, the backup is redundant by the time the next write happens, and the cost (one extra file por write, scanned on every `eza --git` and every `fd` walk) outweighs the benefit. The 6 sub-comando opt-in is the smallest change that closes the API parity hole (GAP-2026-013 Problem C) without making any sub-comando a regression for users who don't need backups. The helper idempotence and the `tracing::warn!` on non-NotFound errors preserve the L4 "the operation completed, the cleanup is advisory" principle that ADR-0028 established for the WAL sidecar.

- **Alternativas consideradas**:
  1. **Keep `cleanup_old_backups_in` count-based prune, add `--keep-backup` only.** Rejected: the count-based prune still leaves the first 5 backups forever and the 6th-and-after only on a 5-more-writes trigger. A user who writes 4 times never sees any cleanup. The metric (`Backups órfãos após sucesso: 0`) exige deletion-on-success, not deletion-on-overflow.
  2. **Add a separate `prune-backups` sub-comando and let the operator executar it on a schedule.** Rejected: introduces a daemon-adjacent operational burden (schedule, monitor, alarm) for a problem that pode ser solved inline. The `regras-rust-cli-one-shot §Daemon Proibido` invariante forbids anything that would exige a long-executando processo for cleanup.
  3. **Age-based prune: `--max-backup-age <SECONDS>`.** Rejected: a 0-second age is equivalent to "delete immediately" (this ADR); a non-zero age reintroduces the "when do they go away" question that this ADR resolves by making the answer "after the write that created them succeeds". Subsumed by deletion-on-success.
  4. **Force `--backup` to imply `--keep-backup` (no opt-out).** Rejected: violates the project's "every flag is opt-in" convention. The 2026-06-15 incident is the precedent for adding safety flags, not the precedent for removing operator choice.

- **Gatilho para revisitar**: If a user reports that a backup foi needed after success and they did not pass `--keep-backup` (e.g. a forensic timeline analysis that exige the post-success state to be reversible), revisit the padrão. If the `delete_backup_quietly` failure rate on real workloads exceeds 0.1% (measured através de the new `tracing::warn!` on non-NotFound), add a `--strict-cleanup` flag that propagates cleanup errors as non-zero saída. If a future sub-comando (e.g. `seguro-write`, `move`) needs the same behavior, extend the flag list rather than introducing a sub-comando-specific opt-in.

## Addendum (v0.1.22 — revisit of Alternative 2)

The v0.1.22 audit identified a context this ADR did not anticipate: **legado `.bak.YYYYMMDD_HHMMSS` siblings created under v0.1.20 retention semantics**. Under v0.1.20, `cleanup_old_backups_in` retained the 5 most recent backups por file; an operator who upgraded from v0.1.20 to v0.1.21 inherits one `.bak.<timestamp>` sibling por `--backup` write they ever issued under the old retention semantics. The deletion-after-success invariante in the body of this ADR endereça the **forward-facing** problem (new writes não leave persistent trash) but leaves the **backward-facing** problem (legado v0.1.20 backups persist) without a removal path.

Alternative 2 of this ADR ("Add a separate `prune-backups` sub-comando and let the operator executar it on a schedule") foi rejected at the time on the grounds that it introduced a daemon-adjacent operational burden. v0.1.22 revisits that decision with a narrower escopo: the `prune-backups` sub-comando is **explícito** (the operator runs it on purpose, not as a side effect of another sub-comando), **seguro** (padrões to `--dry-executar true` to previne accidental data loss), and **componível** (uses the same `--workspace` jail and the same envelope NDJSON as the rest of the CLI). The operator-driven one-shot invocation satisfies the `regras-rust-cli-one-shot §Daemon Proibido` invariante that foi the original reason for rejection — there is no schedule, no background processo, no implícito cleanup.

The full rationale for the v0.1.22 sub-comando is documented in ADR-0040 (`docs/decisions/0040-prune-backups-sub-comando.md`). The two ADRs são complementary: ADR-0038 governs the ciclo de vida of backups created by new writes (deleted after success by padrão); ADR-0040 governs the explícito cleanup of pre-v0.1.21 backup siblings.

No retroactive automatic cleanup is introduced. An operator who upgrades from v0.1.20 to v0.1.22 must executar `prune-backups --max-age 0 --dry-executar false` explicitamente to clean up legado backups. This is intentional: the v0.1.20 backups foram created under different semantics (implícito count-based retention), and silently deleting them on `init` or on first write of the new version would be a data-loss bug, not a cleanup.
